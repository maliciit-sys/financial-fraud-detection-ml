1. Introduction

Financial fraud represents a critical threat to the global economy, costing billions of dollars annually and eroding consumer trust in financial institutions. Traditional rule-based fraud detection systems struggle to keep pace with increasingly sophisticated fraud schemes. This project leverages artificial intelligence and neural networks to create an adaptive, learning-based fraud detection system capable of identifying complex patterns that evade conventional detection methods.

The system analyzes transaction characteristics including amount, velocity, geographical patterns, spending behavior, and device information to assess fraud risk in real-time. By processing historical fraud patterns through deep learning, the model learns to distinguish legitimate transactions from fraudulent ones with high accuracy.

2. Project Objectives

2.1 Primary Objectives

  • Develop a neural network model capable of detecting fraudulent transactions with high recall
  • Handle severe class imbalance (26.8:1 ratio) through appropriate loss weighting
  • Achieve real-time prediction capability for production deployment
  • Minimize false negatives (missed frauds) while maintaining acceptable false positive rates

2.2 Technical Objectives

  • Design and implement a multi-layer perceptron architecture using PyTorch
  • Apply proper data preprocessing including normalization and encoding
  • Implement early stopping to prevent overfitting
  • Evaluate model performance using precision, recall, F1-score, and ROC-AUC metrics

7.2 Feature Set

The model uses 13 features derived from transaction data:

  • Numeric Features: amount, time_since_last_transaction, spending_deviation_score, velocity_score, geo_anomaly_score

7.3 Data Preprocessing Pipeline

  1. Missing Value Imputation:
    • fraud_type: Filled with "Unknown" category
    • time_since_last_transaction: Filled with median value
  2. Categorical Encoding: LabelEncoder applied to 8 categorical features
  3. Feature Scaling: StandardScaler normalization (mean=0, std=1) for numeric features
  4. Data Splitting: Stratified split maintaining fraud ratio
    • Training: 70% (3,502,000 samples)
    • Validation: 15% (748,000 samples)
    • Test: 15% (750,000 samples)

8.2 Root Cause Analysis

The baseline model's failure to detect fraud stems from the severe class imbalance (26.8:1 ratio). The standard Binary Cross-Entropy loss function treats all prediction errors equally, leading the model to optimize for overall accuracy by simply predicting the majority class (non-fraud) for all transactions.

Mathematical Explanation: Predicting "all non-fraud" yields 96.41% accuracy with zero learning effort. The model has no incentive to learn fraud patterns when it can achieve high accuracy by ignoring the minority class entirely.

8.3 Proposed Optimization Strategy

The solution involves implementing BCEWithLogitsLoss with pos_weight=26.8, which assigns 26.8x higher penalty for missing fraud cases compared to false alarms. This forces the model to prioritize fraud detection.

Trade-off: This optimization will increase false positive rate (legitimate transactions flagged as fraud) but dramatically improve fraud detection recall, which is the primary objective in fraud detection systems.

9. Key Visualizations & Insights

9.1 Confusion Matrix Analysis

The confusion matrix clearly illustrates the baseline model's complete failure to identify fraudulent transactions. All 26,933 fraud cases in the test set were incorrectly classified as non-fraud, resulting in 100% false negatives.

9.2 ROC Curve

The ROC-AUC score of 0.500 indicates the model performs no better than random guessing. The ROC curve overlays perfectly with the diagonal line, confirming the model has not learned to discriminate between fraud and non-fraud cases.

9.3 Feature Correlation Insights

The correlation matrix reveals that the numeric features (amount, time_since_last_transaction, spending_deviation_score, velocity_score, geo_anomaly_score) are largely independent, suggesting each contributes unique information for fraud detection. This validates the feature engineering approach.

9.4 Class Distribution

Visualizations confirm the severe imbalance with fraud representing only 3.59% of transactions. The pie chart and bar charts clearly demonstrate why specialized techniques (weighted loss) are essential for this problem.

9.5 Training Curves

The training loss decreased from 3.631 to 3.591 over 4 epochs, while validation loss remained stable at 3.590. Early stopping prevented overfitting, though the high absolute loss values indicate poor model learning under standard BCE loss.

10. Recommended Improvements

10.1 Immediate Optimizations (Priority 1)

  1. Weighted Loss Implementation: Apply BCEWithLogitsLoss with pos_weight=26.8 to address class imbalance
  2. Threshold Tuning: Adjust decision threshold from 0.5 to optimize precision-recall trade-off based on business requirements
  3. Extended Training: Allow more epochs (10-20) with early stopping to ensure convergence

10.2 Advanced Enhancements (Priority 2)

  1. Architecture Modifications:
